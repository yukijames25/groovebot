# GrooveBot — 歌声・ハミングにリアルタイムで“ノる”上半身ロボット

> アカペラの歌声やハミングを聴いて、テンポと盛り上がりに合わせて上半身でグルーヴするロボットの研究開発プロジェクト。

GitHub: https://github.com/yukijames25/groovebot

---

## 一言で

「人が鼻歌を歌うと、ロボットがその場でノリ始める」という体験を、**伴奏なし・リアルタイム**で成立させることを目指した個人R&Dプロジェクト。音楽情報処理（MIR）・ロボット制御・人とロボットの引き込み（entrainment）の交差点に位置する。

## なぜ難しく、なぜ面白いか

- **入力がアカペラ／ハミング**。伴奏のある楽曲と違い、和声も打楽器もない単旋律から「いつ・どれくらいノるか」を取り出す必要がある。
- **リアルタイム性**。聴いてから動くまでの遅延が体験を殺す。知覚と制御を分離し、低遅延で回す設計が要る。
- **身体非依存**。自作3Dプリントロボット／Pepper／シミュレータなど、どの“体”にも載せ替えられること。

研究的には、音楽駆動のロボットダンス、リズム的引き込み、オンラインビート追跡、音楽感情推定、music→dance生成といった複数領域の知見を踏まえて設計している。

## システム設計

「いつ動くか（拍・テンポ）」と「どう動くか（グルーヴの質）」を分け、**頭脳と身体を完全に分離**したインターフェース駆動の構成を採った。

```
歌声/ハミング
   │
   ▼
[ 知覚: ReferenceAligner ]   ← 選曲した既知曲の参照(SongReference)へ生歌声を整合
   │   (拍グリッド・構造・盛り上がりを参照から継承)
   ▼
 GrooveContext  (拍位相 / テンポ / 構造 / arousal)
   │
   ▼
[ グルーヴ生成器 ]           ← 現状ルールベース、将来は学習生成
   │
   ▼
 JointCommand                ← 関節指令(URDF関節上限でクランプ)
   │
   ▼
[ RobotBackend ]             ← sim / Pepper / 自作機 を差し替え可能
   │
   ▼
 ロボットの動き
```

- **Orchestrator** が制御ループ（30–50Hz）を回し、重い知覚処理から分離。
- 公開インターフェース（`GrooveContext` / `JointCommand` / `RobotBackend`）を固定契約として保ち、頭脳が特定のシミュレータやロボットSDKに依存しないようにしている。
- 関節出力は常にURDFの関節上限へクランプ（安全要件）。

## 技術的な意思決定のハイライト

このプロジェクトで最も価値があるのは、コードそのものより**証拠に基づいて方針を変えた判断**だと考えている。

### 1. 盲目ビート追跡から「参照アライメント」への転換

当初はライブ音声から直接ビートを推定する方針（BeatNet/madmom）で進めたが、検証の過程で **madmom が現代のColab環境（Python 3.12 / numpy 2.x）で構造的に動かない**ことが判明した。ここで小手先の回避を続けるのではなく、**ターゲット用途（カラオケ＝既知曲）の性質を使って問題を作り替えた**：

- ユーザは曲を選ぶ → その曲の参照（拍・ダウンビート・構造・特徴量）を持てる。
- ライブの歌声を、その参照タイムラインへ**整合（align）**させる。
- すると、不安定な盲目追跡に頼らず**信頼できる拍グリッドと構造を継承**でき、先読み的なグルーヴすら可能になる。

盲目追跡は任意のフォールバックへ降格し、依存の腐敗もスコープ外へ追い出した。

### 2. 評価ファースト

機能を足す前に **mir_eval ベースの評価ハーネス**を用意し、すべてのマイルストーンを同一指標（F-measure / CMLt / AMLt / リアルタイム係数）で測れるようにした。良い数字も悪い数字も、同じ物差しで判断する。

## 主要な結果

### M1 — リアルタイム・グルーヴループ（完了）

知覚から分離した30–50Hzの制御ループ、型付きインターフェース、ルールベースのグルーヴ生成、URDF関節上限クランプまでを実装。

### M0′ Tier 1 — 参照アライメントの実現性検証（強い結果）

参照（ボーカル＋拍グリッド）に既知のテンポ変動を与えた合成クエリを、オフラインDTWで整合し、拍グリッドの復元精度を測定（GTZAN系7ジャンル×複数曲×5倍率 = 70ラン）。

| 指標 | 値 |
|---|---|
| F-measure（全体） | **0.983** |
| CMLt | 0.984 |
| AMLt | 0.984 |
| リアルタイム係数 | 0.017（約60倍速） |

特筆すべきは **CMLt ≈ AMLt（半・倍テンポの取り違えがゼロ）** という点。これは盲目ビート追跡が最も苦手とする領域であり、参照アライメント方式の**構造的な強み**がそのまま数字に現れた。

> 注: Tier 1 は自己ワープのため楽観値であり、「機構が正しく動く・テンポ変動に強い」ことの確認。別演者の歌唱・鼻歌での難しさは Tier 2 で測る、と最初から切り分けている。

### M0′ Tier 2 — 実演奏での頑健性（進行中）

実アマチュア歌手の独立演奏で評価するため、ライセンス済みデータセット（DAMP系）を取り込む経路を実装。**歌唱はchroma経路、鼻歌はpitch経路**の二系統で同一指標により評価する。

現在は、代表性の高いポップ楽曲データ（DAMP-VSEP）のアクセス許諾待ちの段階。手元で先行入手できた賛美歌データ（Amazing Grace, 17,582独立演奏）での検証を通じて、評価パイプラインを方法論的に堅牢化した（下記）。

## このプロジェクトが示すもの

- **誠実な実証主義**: 実データでの初回スコアが低かった際、結果を鵜呑みにも棄却もせず、「タスクが本当に難しいのか／計装のアーティファクトか」を切り分ける**構造化された診断**を実施。原因（演奏ごとの起点オフセット、特徴量のスパース非対称によるDTW退化）を特定し、**正解データ（GT）で出力を調整しない＝テストリークを避ける**形でのみ修正した。
- **判断の規律**: 外れ値（遅いルバート賛美歌は、整合が難しい上にそもそもグルーヴ対象でない二重に非代表的なケース）への過剰最適化を意図的に打ち切り、代表値はDAMP-VSEPで取る、と線を引いた。
- **システム思考**: 身体非依存のインターフェース設計、知覚と制御の分離、安全クランプ。
- **証拠駆動の方針転換**: madmom行き止まりからの参照アライメント主軸化。
- **ライセンス／著作権の衛生**: ライセンスデータは一切公開リポジトリへコミットせず、ローカル限定・再配布禁止を徹底。
